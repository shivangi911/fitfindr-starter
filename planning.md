## Tools

### Tool 1: search_listings

**What it does:**
`search_listings` searches the mock secondhand listings dataset for items that match the user's requested clothing description, optional size, and optional maximum price. It filters listings first, then scores them by keyword overlap so the most relevant matches appear first.

**Input parameters:**
- `description` (str): The item or style the user is searching for, such as `"vintage graphic tee"` or `"black combat boots"`.
- `size` (str | None): The user's requested size, such as `"M"`, `"8"`, or `"XXS"`. If `None`, the tool does not filter by size.
- `max_price` (float | None): The highest price the user wants to pay. If `None`, the tool does not filter by price.

**What it returns:**
A list of matching listing dictionaries sorted by relevance. Each dictionary may contain fields such as `id`, `title`, `description`, `category`, `style_tags`, `size`, `condition`, `price`, `colors`, `brand`, and `platform`. If no listings match, it returns an empty list `[]`.

**What happens if it fails or returns nothing:**
If no listings match the query, the tool returns `[]` instead of raising an exception. The agent checks for this empty list, stores a helpful error message in the session, and stops early instead of calling `suggest_outfit` with missing item data.

---

### Tool 2: suggest_outfit

**What it does:**
`suggest_outfit` takes the selected thrift listing and the user's wardrobe, then asks the LLM to create one or two complete outfit ideas. If the wardrobe has items, the tool uses those specific pieces; if the wardrobe is empty, it gives general styling advice.

**Input parameters:**
- `new_item` (dict): The selected listing from `search_listings`, including details like title, price, platform, colors, style tags, condition, and description.
- `wardrobe` (dict): A wardrobe object with an `items` key containing the user's existing wardrobe pieces. It may be empty for a new user.

**What it returns:**
A non-empty string with one or two complete outfit suggestions. The suggestion should include what to pair with the new item, the overall style vibe, and a practical styling detail.

**What happens if it fails or returns nothing:**
If the wardrobe is empty, the tool still returns a useful general outfit suggestion. If the LLM call fails, the tool returns a safe fallback suggestion based on the item details so the agent can continue.

---

### Tool 3: create_fit_card

**What it does:**
`create_fit_card` turns the outfit suggestion and selected listing into a short, shareable caption that sounds like an Instagram or TikTok outfit post.

**Input parameters:**
- `outfit` (str): The outfit suggestion generated by `suggest_outfit`.
- `new_item` (dict): The selected thrift listing used in the outfit.

**What it returns:**
A 2–4 sentence caption that mentions the item title, price, platform, and outfit vibe in a casual way.

**What happens if it fails or returns nothing:**
If the outfit string is empty or missing, the tool returns `"Unable to create fit card because outfit information is missing."` rather than crashing. If the LLM call fails, the tool returns a simple fallback caption using the item details.

## Planning Loop

**How does your agent decide which tool to call next?**

The agent starts by creating a session dictionary for the current user interaction. It parses the user's natural language query with simple regex and string cleaning to extract three values: `description`, `size`, and `max_price`. These parsed values are stored in `session["parsed"]`.

The first tool call is always `search_listings(description, size, max_price)`, because the agent needs an item before it can suggest an outfit. After search returns, the agent checks `session["search_results"]`. If the result list is empty, the agent sets `session["error"]` to a helpful message and returns early. It does not call `suggest_outfit` or `create_fit_card`.

If search returns at least one result, the agent selects the top result with `selected_item = results[0]` and stores it in `session["selected_item"]`. Then it calls `suggest_outfit(selected_item, wardrobe)` and stores the returned text in `session["outfit_suggestion"]`. If that text is missing, the agent stores an error and returns early. Otherwise, it calls `create_fit_card(outfit_suggestion, selected_item)`, stores the result in `session["fit_card"]`, and returns the final session.

---

## State Management

**How does information from one tool get passed to the next?**

The agent uses one session dictionary as the single source of truth for the interaction. The session stores the original query, parsed search parameters, search results, selected item, wardrobe, outfit suggestion, fit card, and any error message.

State flows like this:
1. The raw user query is stored in `session["query"]`.
2. Parsed values are stored in `session["parsed"]`.
3. `search_listings` results are stored in `session["search_results"]`.
4. The top result is stored in `session["selected_item"]`.
5. `session["selected_item"]` and `session["wardrobe"]` are passed into `suggest_outfit`.
6. The returned outfit text is stored in `session["outfit_suggestion"]`.
7. `session["outfit_suggestion"]` and `session["selected_item"]` are passed into `create_fit_card`.
8. The final caption is stored in `session["fit_card"]`.

The user does not need to re-enter information between tool calls because each step reads from the session.

## Error Handling

For each tool, describe the specific failure mode you're handling and what the agent does in response.

| Tool | Failure mode | Agent response |
|------|-------------|----------------|
| search_listings | No results match the query | Store an error message like: `"I couldn't find listings for that exact request. Try a broader description, a higher budget, or leaving out the size filter."` Return early without calling the other tools. |
| suggest_outfit | Wardrobe is empty | Return a general styling suggestion using common wardrobe items instead of crashing or returning nothing. |
| create_fit_card | Outfit input is missing or incomplete | Return `"Unable to create fit card because outfit information is missing."` instead of raising an exception. |

## A Complete Interaction (Step by Step)

Write out what a full user interaction looks like from start to finish — tool call by tool call. Use a specific example query.

**Example user query:** "I'm looking for a vintage graphic tee under $30. I mostly wear baggy jeans and chunky sneakers. What's out there and how would I style it?"

**Step 1:**
The agent creates a session and parses the query. It extracts:
- `description = "vintage graphic tee"`
- `size = None`
- `max_price = 30.0`

Then it calls:
`search_listings("vintage graphic tee", size=None, max_price=30.0)`

If the tool returns matching listings, the agent stores them in `session["search_results"]`.

**Step 2:**
The agent selects the top result:
`session["selected_item"] = session["search_results"][0]`

For example, this might be a faded band tee from Depop for $22. The agent then calls:
`suggest_outfit(session["selected_item"], session["wardrobe"])`

The outfit tool uses the selected item and wardrobe to suggest a complete outfit, such as pairing the tee with baggy jeans and chunky sneakers.

**Step 3:**
The agent stores the outfit suggestion in `session["outfit_suggestion"]`. Then it calls:
`create_fit_card(session["outfit_suggestion"], session["selected_item"])`

This creates a short caption for the outfit.

**Final output to user:**
The user sees three panels:
1. The top listing found, including title, price, platform, size, condition, and description.
2. A complete outfit idea using the item.
3. A casual shareable fit card caption.

If no listings were found in Step 1, the user instead sees a message explaining that no listings matched and suggesting that they broaden the description, increase the budget, or remove the size filter.
